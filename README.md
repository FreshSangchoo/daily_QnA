# Daily Q&A

감성적인 기록을 남기는 하루 1문 1답 서비스

매일 하나의 질문이 주어지고, 사용자는 그 질문에 대한 답변과 함께 오늘의 사진 1장, 짧은 한 줄 기록을 남기는 구조
가장 중요한 경험은 1년 뒤 같은 질문이 다시 주어졌을 때, 작년에 쓴 나의 답변을 돌아보며 올해의 답변을 새로 기록하는 흐름

- 매일 질문 1개 제공
- 질문에 대한 답변 작성 및 저장
- 답변과 함께 오늘의 사진 1장과 한 줄 기록 저장
- 1년 뒤 같은 질문 재노출
- 작년 답변을 보며 올해 답변 재작성 가능
- 질문에 따라 다른 사람들의 답변 통계를 선택적으로 확인 가능

## 주요 기능

### 1. 오늘의 질문

- 하루에 하나의 질문 제공
- 사용자가 부담 없이 기록할 수 있도록 짧고 명확한 질문 구성

### 2. 답변, 사진, 한 줄 기록

- 질문에 대한 답변 작성
- 오늘의 사진 1장 저장
- 오늘을 짧게 남길 수 있는 한 줄 기록 작성

### 3. 1년 뒤 다시 만나는 질문

- 동일한 질문의 1년 뒤 재등장
- 이전 답변과 현재 답변 비교 가능
- 시간의 흐름에 따라 생각과 감정의 변화 체감 가능

### 4. 선택형 통계

- 모든 질문에 통계를 노출하지 않는 방식
- 객관적이거나 가벼운 질문에서만 다른 사람들 답변 Top 3 보기 같은 버튼으로 선택적 노출
- 감성적이거나 서사적인 질문에서는 통계를 기본 제공하지 않는 방향

## 질문 유형

질문 성격에 따른 내부 운영 분류

### 1. 통계형 질문

예시:

- 좋아하는 음식은?
- 가장 편한 계절은?
- 아침형 인간인가, 저녁형 인간인가?

특징:

- 답변이 짧고 분류가 쉬운 질문
- 다른 사람들의 답변 통계를 보여줘도 분위기를 해치지 않는 유형
- Top 3 보기 버튼을 통한 선택적 공개 방식

### 2. 회고형 질문

예시:

- 요즘 자주 드는 생각은?
- 최근 나를 가장 웃게 만든 순간은?
- 오늘의 나는 어떤 기분이었나?

특징:

- 감정과 기억, 생각을 돌아보는 질문
- 통계보다 개인의 기록 경험이 더 중요한 유형
- 사진과 한 줄 기록이 더 잘 어울리는 구성

### 3. 비교형 질문

예시:

- 지금의 내가 가장 중요하게 생각하는 것은?
- 요즘 나를 버티게 해주는 것은?
- 올해 나는 무엇을 배우고 싶은가?

특징:

- 1년 뒤 다시 답했을 때 변화가 더 잘 드러나는 질문
- 작년 답변과 올해 답변 비교가 핵심인 유형

## 이 서비스의 특징

- 단순한 Q&A 앱이 아니라 시간이 지나며 의미가 커지는 구조
- 기록의 양보다 다시 돌아왔을 때의 감정이 더 중요한 서비스 경험
- 작년의 나와 올해의 나를 비교하는 구조를 통한 뚜렷한 서비스 정체성
- 사진과 한 줄 기록을 통해 그날의 분위기까지 함께 보존하는 방식

## 사용자 흐름

### 기록하는 날

1. 오늘의 질문 확인
2. 질문에 대한 답변 작성
3. 사진 1장 첨부
4. 한 줄 기록 작성
5. 저장

### 1년 뒤 같은 질문이 돌아온 날

1. 같은 질문 재확인
2. 먼저 올해 답변에 대한 생각 정리
3. 필요 시 작년 답변 펼쳐보기
4. 올해의 답변, 사진, 한 줄 기록 작성
5. 과거와 현재의 답변 비교

## 통계 노출 방식

통계는 비어 있는 순간을 가볍게 채워주는 보조 장치

- 작년 내 답변이 없을 때 특히 유용한 요소
- 질문이 객관적이거나 가벼운 경우에만 사용하는 방식
- 기본 화면에 항상 노출하지 않는 구조
- 사용자가 버튼을 눌렀을 때만 펼쳐 보여주는 형태

초기 데이터 전략:

- 초반에는 실제 유저 데이터가 부족하므로 운영용 시드 데이터 활용
- 완전한 가짜 데이터처럼 보이게 만들기보다 자연스러운 예시 응답 기반으로 시작
- 이후 실제 사용자 응답이 쌓이면 점진적으로 대체하는 전략

## MVP 범위

초기 버전에서 집중할 기능 범위

- 하루 1개의 질문 제공
- 질문 답변 작성 및 저장
- 사진 1장 업로드 및 저장
- 한 줄 기록 작성 및 저장
- 1년 뒤 같은 질문 다시 노출
- 작년 답변 비교
- 일부 질문에 한해 선택형 통계 제공

## 확장 기능

MVP 이후 확장 가능한 기능 후보

- 1년 전 오늘 알림
- 질문 스킵 또는 하루 1회 교체
- 월별, 연도별 기록 모아보기
- 답변 공개 범위 설정
- 익명 통계 포함 동의 설정
- 작년 답변 가리기 후 작성 완료 뒤 공개
- 나만의 질문 저장

## 기술 구성

이 프로젝트에서 사용할 기술 조합

- 프론트엔드: React + TypeScript + Vite
- 프론트 배포: Vercel
- 백엔드: Java Spring Boot
- 백엔드 배포: AWS EC2
- 데이터베이스: PostgreSQL on AWS RDS
- 이미지 저장: AWS S3

### 기술 선택 배경

- React + TypeScript + Vite
  질문 카드, 답변 작성, 작년 답변 비교, 사진 업로드 흐름 같은 상태 중심 UI를 빠르고 가볍게 구현하기에 적합한 구성
- Vercel
  프론트 배포의 단순함, HTTPS, 도메인 연결, 배포 자동화 측면에서 유리한 환경
- Spring Boot
  질문 제공, 답변 저장, 1년 뒤 재노출, 통계 API, 인증 등 서버 기능 구현에 적합한 백엔드 프레임워크
- PostgreSQL
  사용자, 질문, 답변, 사진 메타데이터, 통계 관련 데이터를 관계형으로 관리하기에 적합한 데이터베이스
- AWS S3
  사용자가 업로드한 사진 저장과 이후 확장에 적합한 스토리지

## 배포 구조

기본 배포 구조

1. 사용자가 Vercel에 배포된 프론트엔드에 접속
2. 프론트엔드가 EC2의 Spring Boot API 호출
3. 백엔드는 질문, 답변, 기록 데이터를 RDS PostgreSQL에 저장
4. 사진은 S3에 업로드하고, DB에는 파일 URL 또는 키만 저장

이 구조의 의미:

- 프론트는 빠르게 개발하고 배포할 수 있는 환경
- 백엔드, DB, 파일 저장은 AWS 기반으로 실전 구조 경험 가능
- 초기 개발 속도와 클라우드 학습 경험을 균형 있게 가져갈 수 있는 조합

## 데이터 저장 방식

이 서비스는 사용자 입장에서 진입 장벽이 낮아야 하지만, 동시에 기록이 장기적으로 보존되어야 하는 구조
그래서 데이터 저장 방식은 두 가지 시나리오를 모두 고려할 필요

### 1. 비로그인 사용자

- 로그인 없이 바로 사용 가능
- 데이터는 해당 브라우저 또는 해당 기기 로컬 저장소에 저장
- 빠르게 시작할 수 있고 부담이 적은 방식

장점:

- 가입, 로그인 과정 없이 바로 기록 가능
- 개인 기록 서비스 특성상 가볍게 시작하기 좋은 흐름
- 초기 사용자 이탈을 줄일 수 있는 구조

한계:

- 브라우저를 바꾸거나 기기를 바꾸면 기록 확인 불가
- 캐시 삭제나 로컬 데이터 삭제 시 기록 유실 가능성
- 장기 보관 관점에서는 불안정한 저장 방식

### 2. 로그인 사용자

- 계정 생성 후 로그인 시 서버에 기록 저장
- 다른 컴퓨터나 다른 기기에서도 같은 계정으로 기록 조회 가능
- 장기적으로 안정적인 기록 보관 가능

장점:

- 기기 변경 후에도 기록 유지 가능
- 1년 뒤 같은 질문을 안정적으로 다시 보여주기 좋은 구조
- 사진과 답변 데이터를 안전하게 관리하기 쉬운 방식

한계:

- 초반 진입 장벽이 높아질 가능성
- 단순 기록 서비스를 기대한 사용자에게 부담으로 느껴질 가능성

## 현재 서비스 방향

현재 서비스에서 생각하고 있는 기본 방향

- 기본은 비로그인으로도 사용 가능
- 사용자가 원할 때 로그인해서 기록을 계정에 연결할 수 있게 하는 구조

기록 서비스의 첫 경험은 가볍게 시작하고, 장기 보관과 동기화가 필요해질 때 로그인 가치를 느끼게 만드는 방식

기본 흐름:

- 첫 사용은 로그인 없이 가능
- 로컬 저장으로 바로 기록 시작
- 일정 시점에 로그인하면 기존 로컬 기록을 계정으로 이전하거나 연동

이 방향의 장점:

- 초반 진입 장벽이 낮은 구조
- 기록이 쌓인 뒤 로그인 필요성을 자연스럽게 느끼게 되는 흐름
- 서비스 특성상 감성적이고 개인적인 첫 경험을 해치지 않는 방식

- 로컬 기록과 서버 기록 병합 방식
- 로그인 유도 시점
- 익명 통계 포함 여부에 대한 동의 방식
- 비로그인 상태에서의 사진 저장 지원 범위
